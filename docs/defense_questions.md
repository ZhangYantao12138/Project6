# 答辩可能被问到的问题

---

## 一、架构设计类

**Q: 为什么用三个节点？两个不够吗？**
- node0 专门跑调度控制面（Scheduler 对延迟敏感），与业务任务混在一起会导致调度抖动
- node1 是 I/O 密集型（大量 HTTP 下载），node2 是计算密集型（PIL 解码、聚合），两者混跑会互相抢资源
- 三节点是职责隔离，不是为了"加机器"

**Q: node0 挂了会怎样？**
- Scheduler 和 Webserver 都在 node0，node0 宕机后已在运行的任务（node1/node2 的 worker）会继续跑完，但不会触发新任务
- Redis broker 也在 node0，宕机后 worker 无法接收新任务
- 这是当前设计的单点故障，可以通过 Airflow HA（多个 Scheduler + 外部 Redis 集群）改善，但本项目规模下不必要

**Q: 为什么不用 Spark？**
- 本项目是周期性批量采集（每 2/10 分钟一批），不是持续流，不需要 Spark Streaming
- 每批数据量小（几百张图），Spark 的启动开销反而比实际计算时间长
- Airflow + Celery 对周期性任务的调度和依赖管理更直接

**Q: 为什么 ingest 和 processing 分两个 Queue，而不是一个？**
- 防止采集任务（I/O 密集）和计算任务（CPU 密集）争抢同一节点资源
- Queue 路由确保 ingest 只跑在 node1、processing 只跑在 node2，资源边界清晰
- 如果未来加节点，只需让新节点监听对应 queue 即可水平扩展

## 二、数据模型类

**Q: 为什么用 MongoDB 而不是 PostgreSQL？**
- 摄像头元数据字段异构（不同摄像头有不同地理字段缺失），文档模型无需 migration 就能扩展新字段
- `raw_captures` 每次写入约数百条，MongoDB 的写入吞吐足够，且聚合管道原生支持 `camera_hourly_summary` 的 group-by 统计
- 如果强调"如果用 PG 会怎样"：可以用，但摄像头地理字段的可选性需要大量 nullable 列，或用 JSONB，本质上退化成半结构化存储

**Q: 图像为什么不直接存 MongoDB，而是放 MinIO？**
- 图像是二进制大文件（50–200KB），MongoDB 存 BSON 有 16MB 单文档限制，更重要的是文档数据库不适合存大 blob
- MinIO 的对象存储天然按路径分区（`camera_id/date/hour/`），检索效率高，且兼容 S3 API，未来可迁移至云

**Q: `capture_ts` 和 `ingest_ts` 的区别是什么？为什么要存两个？**
- `capture_ts` 是对齐到调度间隔边界的"计划时间"（如 14:07 下载的图像记录为 14:00），确保跨周期比较一致
- `ingest_ts` 是实际完成下载的时间，用于计算延迟（`ingest_ts - capture_ts`）
- 只存一个的话，要么丢失"计划时间"的对齐语义，要么丢失实际延迟信息

**Q: `derived_features` 里的 `object_key` 为什么要冗余存一份（raw_captures 里已经有了）？**
- 保证特征记录的自完备性：拿到一条 `derived_features` 记录，无需再 join `raw_captures` 就能直接定位原始图像
- 下游 CV 任务或报表只需查 `derived_features`，减少跨集合查询

## 三、数据质量类

**Q: 如何检测"无信号"占位图？**
- 511ny 对没有视频信号的摄像头返回一张固定的灰色占位图，MD5 哈希固定
- 采集时计算每张图的 MD5，与已知占位图哈希比对，匹配则标记 `success=False` 并将摄像头状态改为 `no_feed`

**Q: 重复帧检测的依据是什么？和占位图检测有什么区别？**
- 重复帧：比较同一摄像头相邻两次采集的 SHA-256 哈希，相同说明画面没有变化（摄像头静止或网络缓存）
- 占位图：是特定的"无信号"图，MD5 固定，属于"采集失败"
- 重复帧不一定是失败（夜间无车时画面确实不变），只是质量标记，让下游自己判断是否使用

**Q: `is_corrupted` 检查是在采集时做还是事后做？**
- 事后做（quality_check 阶段）：从 MinIO 重新下载图像并用 PIL 解码
- 这样设计是为了验证"存入 MinIO 的文件是否可用"，而不只是"下载时是否正常"，能发现存储写入损坏的情况

**Q: 如果一个摄像头持续失败，quality_audits 会无限增长吗？**
- 是的，quality_audits 按时间槽写入，每个时间槽一条记录
- 可以通过 TTL 索引（如保留 30 天）或定期归档来控制增长，当前实现未加 TTL，这是一个已知的待优化点

## 四、流水线调度类

**Q: 为什么 `catchup=False`？**
- 图像 API 无历史回放能力，补跑历史时间槽只会请求当前图像，产生错误的时间戳对齐（14:00 的槽实际上存的是当前时间的图像）
- `catchup=True` 在重启后会触发大量补跑任务，全部无效

**Q: 如果采集任务超时（比如 511ny.org 服务慢），会影响下一个调度周期吗？**
- `max_active_runs=1` 确保同一 DAG 不会并发运行，如果上一个 run 还没结束，下一个 run 会等待
- 对于 2 分钟周期的 DAG，如果单次运行超过 2 分钟，会开始积压。可以通过设置 task timeout 和摄像头数量上限来控制

**Q: 标准采集（10 分钟）和优先采集（2 分钟）会不会在同一时间槽产生重复记录？**
- 不会。两条流水线写入 `raw_captures` 时都带有 `pipeline` 字段（`web` 或 `priority`），且时间槽对齐方式不同（10 分钟边界 vs 2 分钟边界）
- 即使同一摄像头出现在两条流水线，`capture_ts` 不同，不会产生主键冲突

## 五、特征提取类

**Q: 为什么特征提取是独立的 DAG，而不是采集 DAG 的下游任务？**
- 解耦：采集频率（2/10 分钟）与特征提取频率（2 分钟）可以独立调整
- 弹性：特征提取可以处理"积压"（过去 2 小时内所有未处理的图像），不依赖特定的触发
- 容错：采集失败不影响特征提取，特征提取失败不影响采集

**Q: 当前是 mock 实现，如果换成真实 CV 模型需要改什么？**
- 只需重写 `feature_extract_mock.py` 中的特征计算函数（`extract_features_from_image(img_bytes)`）
- 数据模型、MongoDB 写入逻辑、任务调度、MinIO 读取全部不变
- 如果真实模型需要 GPU，加一个监听 `cv` queue 的 GPU 节点，把该 DAG 的 queue 改为 `cv` 即可

**Q: `model_id` 和 `model_version` 字段有什么用？**
- 支持多个模型版本共存：不同版本模型的结果存在同一集合，通过 `model_id + model_version` 区分
- 历史结果不被覆盖：更新模型后重跑历史数据，旧结果仍然保留，便于对比模型效果

## 六、可扩展性类

**Q: 如果摄像头数量从 1000 增加到 10000，系统怎么扩展？**
- 增加批次数量（`INGEST_BATCH_COUNT` 环境变量），或增加 node1 节点数量并让新节点也监听 `ingest` queue
- MongoDB 和 MinIO 是水平可扩展的，存储层无需改动
- 唯一需要调整的是 `CAMERAS_PER_BATCH` 和 `INGEST_BATCH_COUNT` 的平衡

**Q: 如果要加一个新的地理区域（比如新泽西州的摄像头），需要改什么？**
- 只需在 `sync_web_cameras.py` 中添加新数据源的 API 调用
- 数据模型（`cameras` 集合）已有 `state`、`county`、`region` 字段，新区域数据直接写入，无需 schema 变更
- 所有下游查询已支持按 `state` 过滤

## 七、项目边界类

**Q: 你们的系统做了交通分析吗？**
- 没有，也不应该做。项目要求是"建流水线支持下游分析"，不是做分析本身
- 我们提供的是：可靠的图像存储、结构化的元数据、质量标记、小时汇总，以及为 CV 预留的 `derived_features` 接口
- 真实 CV 分析（车辆计数、拥堵识别）是下游团队的工作，他们通过我们的 API 获取数据

**Q: 为什么不直接用云服务（S3、RDS）？**
- 项目要求部署在 cluster2（私有集群），不能使用外部云服务
- MinIO 兼容 S3 API，MongoDB 和 PostgreSQL 都是行业标准，迁移到云版本（S3、Atlas、Aurora）无需改代码
